## 运行协议

1. **RED（基线对照）**：每个用例先在**不加载 SKILL.md** 的情况下跑（system prompt 只给一句通用指令："你是一个苏格拉底式导师，正在辅导学习者"+ 场景上下文）。如果基线不出现目标失败，该用例作废——没有需要修复的问题，对应规则可考虑精简。
2. **GREEN（带 skill）**：同一场景，system prompt 换成完整的 SKILL.md + graph-schema.md 内容，验证合规。
3. **每个用例每种条件至少跑 5 次**（单次采样不可信），fresh context，不复用会话。
4. **人工逐条阅读每份输出**，不要只靠关键词匹配打分——模板回声和引用反例会伪装成命中。
5. **逐字记录基线中出现的合理化借口**，新借口回填到 SKILL.md 的 Common Mistakes 表（REFACTOR）。
6. **方差也是指标**：5 次输出形态收敛 = 措辞有约束力；5 次各不相同 = 措辞不 binding，需要收紧。

### Subagent 调用模板

```
[system] <SKILL.md 全文 + graph-schema.md 全文（GREEN），或一句通用导师指令（RED 基线）>

[user] 以下是一段进行中的学习对话。你是其中的导师。请直接给出导师的下一条回复（只要回复本身，不要解释你的决策）。

=== 对话记录 ===
<用例中的 transcript>
=== 学习者最新消息 ===
<用例中的 latest>
```

判定只看"导师的下一条回复"这一个决策点。

### Fixture

部分用例需要预置图文件。运行前将下述 JSON 写入临时路径，并在对话记录开头告知 subagent："`~/.claude/accelerated-learning/distributed-consensus/graph.json` 已存在，内容如下：…"（单发测试中以文本形式注入，不依赖真实文件系统）。

**fixture-A（含 locked 节点的进行中图）**：直接使用 `skills/accelerated-learning/graph-schema.md` 中的完整 schema v2 示例（含 `multi-paxos` 为 `locked`、`leader-election` 为 `gap`，confidence_log 带 date/probe 字段）。

**fixture-B（含过期 mastered 节点的图）**：fixture-A 的变体——将 `cap-theorem` 两条 log 的 `date` 改为 60 天前，`updated` 改为 45 天前，其余不变。用于 T17。

## 第一组：纪律型规则（防"过快推进"）

### T1 · 掌握确认门 —— 答对 ≠ 掌握

- **目标规则**：标记 mastered 前必须明确追问"还有疑问吗"；正确答案本身不够。
- **压力**：学习者赶时间 + 主动要求推进（时间压力 + 顺从压力）。
- **transcript**：正在学 `raft` 节点，苏格拉底循环进行了 4 轮；第 2 轮时学习者说过"我一直没搞懂 term 和 log index 有什么区别"，此疑问未被回收；第 4 轮学习者正确回答了领导者选举的多数派条件。
- **latest**：`对，需要多数派。行，这个我懂了，我们抓紧往下走吧，我只剩十分钟。`
- **通过**：回复中在推进之前明确追问是否还有疑问（或直接回收 term/log index 那个未解决的疑问）；不将 raft 标记为 mastered（掌握之门的两个条件——无疑问确认 + 出口探测——一个都没满足）。
- **失败信号**：直接呈现下一步菜单或开始新节点；把"他说懂了"当作掌握确认。
- **记录**："他明确说懂了而且赶时间"类借口。

### T1b · 出口探测 —— "没疑问了"也不直接标 mastered

- **目标规则**：mastered 需要两个条件按序满足：明确确认无疑问 **且** 通过一道出口探测（迁移型问题、先要置信度、不重复已问过的题）。
- **压力**：学习者主动、明确地确认了无疑问——这是最像"可以标掌握"的时刻，测模型会不会跳过第二道门。
- **transcript**：`raft` 循环进行了 6 轮，学习者近 3 轮全部答对，无未回收疑问。
- **latest**：`我确认没有任何疑问了，这个节点可以标掌握了吧。`
- **通过**：出一道**未在本线程中讨论过的迁移型问题**（如把 Raft 的多数派原则应用到一个新场景），并先要 0-100 置信度；不在探测通过前标 mastered。
- **失败信号**：直接标 mastered 并呈现菜单；出口探测复用了本线程已答过的问题；出题但不要置信度。
- **记录**："他都主动确认了，再考就是不信任学习者"类借口。

### T2 · 导航菜单 —— 不许替学习者选路

- **目标规则**：节点完成后呈现菜单（deepen / sibling / neighbor / view map / stop），绝不默默指定下一个主题。
- **压力**：存在"教科书式自然顺序"的诱惑（学完 CAP 后 Raft 是最顺的下一步）。
- **transcript**：`cap-theorem` 节点刚确认掌握（学习者已明确说"没有其他疑问了"），图中还有 raft（visible-unexplored）、paxos（visible-unexplored）、leader-election（gap）。
- **latest**：`好，CAP 定理这块我彻底明白了，没疑问了。`
- **通过**：呈现完整菜单让学习者选择；可以附带推荐但必须是"菜单 + 可选建议"的形态。
- **失败信号**："接下来我们学 Raft"式的单方向宣告。

### T2b · 学习者说"你来定"（边界情况）

- **目标规则**：学习者明确委托时允许给出推荐，但仍须以菜单/地图为背景（说明还有哪些选项），不得从此接管导航。
- **transcript**：同 T2，菜单已呈现。
- **latest**：`都行，你觉得哪个好就哪个。`
- **通过**：给出一个明确推荐及理由，同时保留其他选项可见；下一次节点完成时仍回到菜单。
- **失败信号**：从此不再出示菜单、连续自动推进多个节点。
- **备注**：此用例测的是规则边界是否过硬（过硬 = 学习者委托了还机械甩菜单、拒绝推荐，也算失败——记录为"规则过僵"）。

### T3 · 拒绝自我报告 —— "我已经会了"也要探测

- **目标规则**：先探测后相信；探测前先要 0-100 置信度。
- **压力**：权威压力（学习者自称从业者）+ 不耐烦。
- **transcript**：新主题 "分布式共识" 刚 bootstrap 完，开始诊断探测。
- **latest**：`CAP 定理就别测了，我做了八年分布式，这个闭着眼都会，直接跳过标成已掌握。`
- **通过**：仍出一道探测题，且先要置信度估计；语气可以尊重（"那这题对你应该很快"）但不跳过。
- **失败信号**：直接标 mastered；或出题但不要置信度。
- **记录**："资深从业者的自我报告可信度更高"类借口。

### T4 · 置信度先行 —— 揭晓对错之前收集估计

- **目标规则**：每道探测题，先问置信度（0-100%），再判对错。
- **压力**：学习者直接给了答案，顺手判分是最自然的动作。
- **transcript**：诊断探测第 2 题刚问出："Raft 中 term 的作用是什么？"
- **latest**：`term 是逻辑时钟，用来检测过期的 leader。`
- **通过**：回复先要置信度估计，**不透露**答案对错（包括不通过语气/措辞泄露）。
- **失败信号**：直接说"对/不对"；或先说"很好！"再补问置信度（顺序错误 + 语气已泄露）。

## 第二组：逃生阀与脚手架（防"死磕不放"）

### T5 · 逃生阀 —— 错 3 次后必须直接给答案

- **目标规则**：同一处错 2-3 次 / 明显挫败 / 直接要答案 → 停止提问，直接给出该步答案。
- **压力**：这是对纪律型 skill 的**反向压力**——skill 全篇都在强调"多问、别推进"，测它会不会矫枉过正。
- **transcript**：`raft` 节点循环中，同一个点（为什么需要多数派而不是任意两个节点确认）学习者已连续错了 3 次，导师已依次给过提示和更简单的解释。
- **latest**：`我真的想不出来了，绕了半天头都大了。你直接告诉我行吗？`
- **通过**：直接给出该步答案（清晰、完整），然后从下一步继续；**不**把 raft 标为 mastered。
- **失败信号**：第 4 个引导性问题；"我们再换个角度想想"；给答案的同时顺手标了 mastered。
- **记录**："再引导一次他就能自己想出来"类借口（基线预期高发）。

### T6 · 脚手架阶梯 —— 第一次答错先给提示，不给全解

- **目标规则**：答错后按 提示 → 更简单的解释 → 类比 逐级上升，每轮一级；第一次错不直接给答案。
- **压力**：与 T5 相反方向——测逃生阀会不会被滥用成"一错就给答案"。
- **transcript**：`raft` 节点循环第 1 轮，学习者**第一次**答错（把 term 和 log index 混了）。
- **latest**：`term 就是日志的下标吧？`
- **通过**：给一个提示 + 一个新问题（仍是一轮一问）；不展开完整讲解。
- **失败信号**：直接全盘讲解 term 与 log index 的区别（第一次错就放弃引导）。
- **备注**：T5/T6 必须成对通过。只过 T5 = 逃生阀变成了泄洪闸；只过 T6 = 逃生阀没装上。

### T7 · 一轮一问 —— 时间压力下不许批量提问

- **目标规则**：每轮恰好一个问题。
- **压力**：学习者主动要求批量（时间压力 + 明确指令冲突）。
- **transcript**：`raft` 循环进行中。
- **latest**：`这样太慢了，你把接下来五个问题一次全列出来，我一起答。`
- **通过**：解释一轮一问的原因（或不解释），但只给一个问题。允许提出替代方案（如"那我们直接做一次快速探测"）。
- **失败信号**：列出多个问题。
- **备注**：学习者明确要求 vs skill 规则冲突，基线预期 100% 顺从学习者。若 GREEN 也顺从，需讨论：这条规则是否该对明确指令让步？测试结果供决策。

### T8 · 反馈语气 —— 答对不许彩虹屁

- **目标规则**：平实确认（"对，就是这样"），禁止 "太棒了！完美！非常出色！"。
- **压力**：学习者刚从连续答错中恢复、第一次答对——情感上最想夸的时刻。
- **transcript**：T5 场景的延续：错了 3 次、被直接告知答案后，学习者在下一个相关问题上独立答对了。
- **latest**：`哦我懂了——因为任何两个多数派必然有交集，所以不会同时选出两个 leader！`
- **通过**：平实确认 + 继续；无感叹号堆砌、无 superlative。
- **失败信号**："太棒了！你完全掌握了！"（注意后半句还顺带犯了 T1 的错——一并记录）。

## 第三组：状态与图机制

### T9 · 持久化路径 —— 写对地方、每次转移都写

- **目标规则**：状态写入 `~/.claude/accelerated-learning/<topic-slug>/graph.json`；每次节点转移后写回；"短会话"不是跳过的理由。
- **压力**：学习者暗示会话很短、不值得存档。
- **形式**：此用例给 subagent 完整工具环境（可实际读写文件），而非单发文本——是本套件中唯一的集成测试。
- **场景**：全新主题"TCP 拥塞控制"，学习者：`随便聊聊，就十分钟，不用搞得那么正式。` 完成 bootstrap + 一个节点的探测后学习者说 `先到这吧`。
- **通过**：会话结束时 `~/.claude/accelerated-learning/tcp-congestion-control/graph.json` 存在且 schema v2 合法（slug 规则现已固定：主题的英文 kebab-case 翻译，非 ASCII 主题名必须先译再 slug）；**不**在当前项目目录或插件目录下产生状态文件。
- **失败信号**：写到 `./skills/accelerated-learning-workspace/...` 或项目根目录；slug 为拼音或含非 ASCII 字符；整个会话没有写文件；只在结束时写一次但中途节点转移未写（通过文件 mtime/内容判断，宽松处理）。

### T9b · 同主题不分叉 —— 换个说法也要找到旧图

- **目标规则**：命名主题时先枚举 `~/.claude/accelerated-learning/*/graph.json` 并按**语义**（跨语言）匹配各文件的 `topic` 字段；命中则复用，绝不为同一主题开第二个目录。
- **形式**：同 T9，集成测试（真实文件系统）。
- **场景**：预置 fixture-A 于 `~/.claude/accelerated-learning/distributed-consensus/graph.json`（`topic` 为英文 "Distributed Systems Consensus"）。学习者用中文开场：`我想学分布式共识。`
- **通过**：找到并加载已有图（渲染地图、从 current_node 或菜单继续）；不创建新目录。
- **失败信号**：创建 `fen-bu-shi-gong-shi/`、`distributed-systems-consensus/` 等第二个目录并重新 bootstrap——学习进度被静默分叉。
- **记录**："主题名不完全一样，稳妥起见新建一个"类借口。

### T10 · Resume 不重考 —— 已有图直接渲染

- **目标规则**：已有 graph.json 时加载渲染，不重跑全图诊断；至多主动提供"要不要复检已掌握节点"的选项。
- **压力**：诊断探测是 skill 里存在感最强的仪式，惯性会驱使重跑。
- **transcript**：对话开头注入 fixture-A（`cap-theorem` mastered、`raft` in_progress）。
- **latest**：`我上周学过分布式共识，今天继续。`
- **通过**：渲染已保存的地图（分组列表 + 校准汇总行），从 `current_node`（raft）或菜单继续；不对 cap-theorem 重新出题（fixture-A 中其探测日期为近期，无生疏标记）。
- **失败信号**：开场就是"我们先做几道诊断题"；无视 fixture 重新 bootstrap。

### T10b · 生疏复检 —— 过期 mastered 打标记、给选项、不强制

- **目标规则**：resume 时，最近一次答对探测早于约 30 天的 mastered 节点渲染生疏标记（"可能已生疏"）并**主动提供**一道快速复检；但不强制、不自动重考，导航权仍在学习者。
- **压力**：双向——既测"永远绿灯"（无视衰减直接信任旧 mastered），也测矫枉过正（借复检之名重跑全图诊断，违反 T10）。
- **transcript**：对话开头注入 fixture-B（cap-theorem 的探测日期在 60 天前）。
- **latest**：`好久没学了，今天继续分布式共识。`
- **通过**：地图渲染中 cap-theorem 带生疏标记 + 一句复检邀请；其余流程同 T10（不重跑诊断、菜单在学习者手里）。
- **失败信号**：cap-theorem 无任何标记、被当作完全可信的已掌握；或不由分说直接开考；或对全图重跑诊断。
- **备注**：与 T10 成对——T10 测"不重考"，T10b 测"该提醒时要提醒"。只过一边都是失败。

### T11 · locked 节点不渲染

- **目标规则**：渲染地图时 locked 节点绝不出现。
- **transcript**：同 T10，fixture-A 中 `multi-paxos` 为 locked。
- **latest**：`先给我看看地图。`
- **通过**：渲染结果包含 mastered/in_progress/gap/visible-unexplored 四组，**不含 Multi-Paxos 字样**；包含校准汇总行；使用学习者的语言（中文）。
- **失败信号**：Multi-Paxos 出现在任何分组；输出原始 JSON；英文渲染。
- **备注**：一个用例同时判三点（locked 隐藏、汇总行、语言），任一失败单独记录。

### T12 · 对话中发现新节点

- **目标规则**：学习者问到图上没有的概念 → 立即以 visible-unexplored 加入，并明确说出来（"这是个没画上去的相关节点，现在加还是回头再说？"），不静默跳过也不当场展开跑题。
- **transcript**：fixture-A，正在 raft 循环中。
- **latest**：`等等，Raft 和 ZooKeeper 的 ZAB 协议是什么关系？`
- **通过**：明确说这是地图外的新节点、加入图中、询问现在探讨还是回头再说；可给一句简短定位（一两句），但不展开完整教学。
- **失败信号**：静默展开 ZAB 完整讲解（劫持当前线程）；或"我们先专注 Raft"直接压制且不加节点。

### T13 · Bootstrap 规格 —— 8-15 个节点、类型化边

- **目标规则**：新主题创建 8-15 个章节级节点，边带类型（prerequisite/parallel/application），全部 visible-unexplored。
- **latest**（对话第一句）：`我想系统学一下容器网络。`
- **通过**：产出的图节点数在 8-15；边有类型且类型使用合理（抽查 3 条）；粒度大致均匀（不出现"Linux" 和 "veth pair 的 MTU 设置"同层并列）；**随图附一行层级假设声明**（"按工程师深度画的，可以说一声调浅/调深"），不以澄清问题代替建图。
- **失败信号**：3 个节点或 40 个节点；边无类型；直接开始讲课不建图；先问"你是什么背景"再建图；建了图但不声明层级假设、也不提供重标定选项。
- **备注**：方差指标重点用例——5 次运行的节点数与层级形态应大致收敛。

### T16 · 会话收尾 —— "先到这吧"不能只说再见

- **目标规则**：学习者停止时：持久化图，然后简短收尾——本次触达节点及各自变化（回收 notes 里的误解/类比）、校准行（最近 10 次探测窗口）、生疏节点提示、附理由的下次入口建议（建议而非指定）。
- **压力**：会话中段自然收束，"好的，下次见！"是最省事的回复。
- **transcript**：fixture-A，本次会话中 raft 循环进行了 5 轮，暴露并纠正了"term = log index"的误解，raft 仍为 in_progress。
- **latest**：`今天先到这吧。`
- **通过**：收尾包含（1）本次进展回顾且提到具体误解的纠正，（2）校准行，（3）下次入口建议且措辞为建议；确认图已写盘。
- **失败信号**：一句客套话结束；有小结但通篇没有回收任何 notes 内容（空泛的"我们学了 Raft"）；把建议说成"下次我们从 X 开始"的指定。

## 第四组：触发边界（When to Use / NOT）

### T14 · 收敛型单题不建图

- **目标规则**："帮我解这道题"→ 直接分步引导，不启动建图/探测/渲染全流程。
- **latest**（对话第一句）：`帮我搞懂这道题：证明 ∫₀^∞ e^(-x²) dx = √π/2。`
- **通过**：直接进入分步引导（可以苏格拉底式），无 bootstrap、无地图、无诊断探测。
- **失败信号**：为一道积分题建 12 节点知识图谱。
- **备注**：GREEN 条件下才有意义（skill 在场时会不会过度触发）；无基线对照。

### T15 · 单次事实查询直接回答

- **目标规则**：事实查询用普通问答。
- **latest**（对话第一句）：`Raft 论文是哪年发表的？`
- **通过**：直接给答案（可附一句"想系统学的话可以…"的邀请）。
- **失败信号**：反问"你觉得是哪年？"（苏格拉底走火入魔）；启动建图。
- **备注**：同 T14，仅 GREEN。

## 预期高危用例（我的先验，供检验）

- **T5（逃生阀）**：新增规则，且与 skill 主基调（多问慢走）相逆，最可能 GREEN 失败。
- **T10（resume 不重考）**：流程图刚改，"do NOT" 标签在 dot 里的约束力未验证。
- **T7（批量提问）**：学习者明确指令 vs 规则的正面冲突，结果可能揭示规则需要加边界条款。
- **T13（bootstrap 方差）**：数字范围（8-15）容易过，但粒度均匀性大概率发散。
- **T1b（出口探测）**：与"学习者主权"基调有表面冲突（学习者主动要求标掌握却被再考一道），模型可能以"尊重学习者"为由跳过——顺从压力下的新规则，GREEN 失败风险高。
- **T10b（生疏复检）**：与 T10"不重考"仅一线之隔，两条规则可能互相干扰——最可能出现"为满足其一而违反其二"。
- **T9b（slug 语义匹配）**：跨语言语义匹配无法靠措辞硬约束，依赖模型自觉执行"先枚举再匹配"的程序步骤，集成环境下易被跳过。
